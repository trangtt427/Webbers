Incorta

Unified Data and Analytics Platform

Improving the way people manage and organize their data

The Incorta platform allows realtime data analytics without the need of data warehousing or manipulation. Despite its utility, data analysts using Incorta were dealing with lackluster file management and tedious workflows. I led the designs for an initiative that empowered data analysts to significantly improve their daily functions.

opportunity

The Villain of this Story: A Poor File Management System

Incorta users have long complained about the product limitations around data support. There were several things missing to help them manage their data, such as organizing files into folders, editing files, and searching. Users couldn't tailor their Incorta environment to their liking.

The current Data page was one of the last pieces that needed to be ported over from Angular to React. The duality of this opportunity meant tackling an engineering initiative and a highly requested user ask in parallel.

research and discoveries

Meet the Incorta Data Analyst

For this project, we were servicing the data analysts (known internally as the "power users"). These users leverage advanced analytical tools to dive into their data, working to create visualizations for their stakeholders. Before any of that happens, data first needs to be brought into their Incorta instance through one of two ways. The first way is by connecting the platform to a database and second, is by uploading data files. Going with the former meant users would have to do some extra upfront work if they're looking to customize their data.

Uncovering a Few User Frustrations

Once a file has been uploaded, a user can't change its name.

The Workaround: change the file's name locally prior to uploading. The Catch: having to remember to do this every time the user uploads a file.

A user can only upload one file at a time.

No workaround. Pure deprivation.

No native search functionality.

The Workaround: cmd+F or ctrl+F. The Catch: a user would scroll down to populate the page with more files. Using the search shortcut would only scan what's currently loaded on the page.

Files are laid out on one level. Folders are not supported.

The Workaround: name related files in a way where they can be near each other once uploaded. The Catch: extra upfront effort needed.

Some semblance of a file directory was already implemented in the newer parts of the software, but users still had trouble getting to where they need. Users complained about the existing breadcrumb in the Content page (the only other page that contained files) because it didn't provide a way to browse through different folder levels.

The current navigation pattern only showed the current file location.

Vision

A Modern File Management System

Based on what we heard from our users, we came up with the following criteria for success.

  1. Ability for users to create/modify folders
  2. Ability for users to upload folders
  3. Ability for users to manage permissions to overwrite
  4. Allow "New" button to combine actions for data file and data source
  5. Implement Folder permissions
  6. Ability for users to move files/folders
  7. Improved file navigation

design

Improving the Breadcrumb Pattern

We looked to a few big names in the storage service space like Google Drive, Onedrive, and Dropbox to see which use cases can be applied to our own. We leaned towards the common breadcrumb with a clear distinction of a user's path location, a pattern that users of most modern day technology would recognize.

Scoping

The proposed solutions were great and all but...

We learned that our Front End and Server teams couldn't commit to some of the asks because resources have been allocated to higher priority projects. The few things they couldn't accommodate in the next two sprints included:

  • Functionality to move folders
  • Merging the upload folders and files actions into one prompt
  • Folder-level permissions

The current library couldn't support the first two features and would require a lot of engineering work to make that happen. The Folder-level permissions also opened a can of worms. We failed to realized its complexities and all the possible scenarios. We decided to move these out of our MVP so that we can prioritize getting the other features delivered on time and buy us more time to carefully think through the permissions.

delivery and impact

Incorta 4.6 and the Big Increase in Productivity

A month after the project kickoff, the Data File enhancements were shipped in the Incorta 4.6 release (check out the release notes). Historically, customers don't usually upgrade right away to the latest version as it does take a bit of work, but they were willing to switch to Incorta 4.6 after being promised the Data File upgrades. The release was met with success and praise from our users. Some of our tenured users reached out with news of them enjoying the new workflows.
