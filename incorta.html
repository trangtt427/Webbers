Incorta

Data File Directory

Product Design | Visual Design

Users of the Incorta Data Analytics Platform were adding large amounts of data files into a single location with limited file management capabilities. I introduced a feature enhancement to store files within a directory, solving a longstanding issue for users— poor data management.

opportunity

The Villain of this Story: A Poor File Management System

The current Incorta Data Files page was one of the few remnants of the outdated Angular framework which the platform was built on a few years ago. One of the outstanding company roadmap items was to port the Data Files page over to React, but enhancing this part of the platform was also a long-awaited user request.

Users have complained about the lack of support for data file management in the Incorta platform. They needed a scalable solution for the increasing amount of files.

The Angular build of the Data Files page

the role of design

I was the lead designer for this feature, working closely with Product Management and solutions architects to find ways to optimize for a better data management.

I met with the solutions architects regularly, and later on, looped in engineering to discuss scope. I led this project from start to finish, including the handoff of high fidelity prototypes.

research and discoveries

Understanding How Incorta Users Manage Data

We learned about the users' workflows and their challenges directly from the interactions they had with Product Management and the solution architects. The end users are data analysts who are responsible for bringing data into their team's Incorta instance. From there, relationships are mapped from these data objects to provide meaningful insights through data visualizations.

The Nuances of Bringing Data Files Into Incorta

There are two ways of bringing data into Incorta. First is by connecting to a database, and second, is through uploading CSV or Excel files. However, if users decided to go with the file route, there was no concept of folders to organize the files to their liking. This was one of several issues that users had with the app.

Pain Points

Files cannot be renamed once uploaded into Incorta
Finding files was difficult as there was no way to search and filter
The "New" button corresponded to different actions depending on the tab that the user was on
Uploading one file at a time was time consuming

Users came up with a workaround to make locating files a little easier

To ensure that relevant files are clumped together in the listing page, users adhered to a naming convention before uploading. For example, files that were used together were named in alphanumeric order. It was extra work upfront, but it allowed files to be easier to organize.

Data Analysts wanted to save time and effort

Users wanted to customize their Incorta instance so that they can spend less time and effort on managing data sources and files. If users can spend less time doing so, they can use that time to create data visualizations that can inform company decisions.

Vision

Our goal was to enhance the Data Files tab so that users can customize and better manage their data.

Criteria for Success

  1. The ability to create folders and rename folders
  2. Uploading folders
  3. Managing permissions to overwrite
  4. Allow "New" button to combine actions for data file and data source
  5. Folder permissions
  6. Move files/folders

design

Ideation

In addition to what users said they wanted, I did some research on a few industry leaders to get an idea on how each of them handled storage. I looked to Google Drive, Dropbox, and Egnyte to name a few. I created wireframes to communicate what the new file structure and functionalities would look like.

Along the way, I realized we needed a better navigation to support the new file structure.

Introducing the concept of folders would mean nesting. Now that users will be able to nest folders, they'll need a way to traverse through the file path. The only part of our app that utilizes somewhat of a directory structure is the Content tab, where dashboards and folders can be grouped into folders. However, the current implementation doesn't provide a way for users to navigate back to folders in a path.

The current navigation pattern only showed the current file location.

Users couldn't see the previous folder and were limited to going back one level. We've had ongoing complaints from customers and internal users about this experience not being intuitive. Users wanted the transparency of knowing their location in the hierarchical structure so they can browse back to higher levels.

I redesigned the breadcrumb component to show the root location, an option to look at multiple folders in that path, the previous folder, and the current folder that the user is in.

Navigation v1

I redesigned the breadcrumb component to allow access to multiple folders in the path of the current file.

In a design sync, we eventually agreed that the title of the page shouldn't be styled like the breadcrumb path. The feedback was that the page title should assume its own identity, thus it was decided to move the title of the page to the next line. It would also be helpful to users if the page title stayed in one spot and not having to jump around depending on how nested the file is.

Navigation v2

I iterated on the navigation so that the current page was separate from the breadcrumb path.

The Product Team aligned with the proposed designs, until we were met with engineering constraints.

In a sync with the server and front-end teams, we found out that some of our tasks would not be able to fit in the upcoming release.

  • The server team cannot support the relocating of folders because it was a huge engineering effort and they were slammed with higher priority tasks.
  • The front-end team is unable to merge the upload folders and files into one prompt (their current tech stack doesn't support that).
  • The server team cannot support folder-specific permissions.

"What if a user doesn't have permission to a certain folder, but has access to the file within it?"

We realized the complexity of folders and permissions after meeting with the teams. As a result of our discussion, we agreed to remove some of the enhancements (moving folders and folder-specific permissions) to include them in the later release. For the first release, we are only attaching permissions to files instead of folders. I updated the wireframes to limit the scope of actions and handed it off to Engineering.

the delivery

A Proper Repository for Data Files, Built in React

Small feature enhancements, huge increase in productivity

We had positive feedback from users, despite not being able to truly treat the Data tab like a directory yet. We were told that as a result of the feature, it made a huge difference in their data management workflow. They also expressed their excitement for the upcoming enhancements in the second phase.

This support for data files was a long-awaited wish for the customers. Moving forward, the breadcrumb design is going to be the standard implementation for the other pages in Incorta whenever applicable. The next phase for this project is currently in the works, which includes folder-specific permissions and moving folders around.

The Takeaway: Thoughts and Lessons

Working on this feature was a reminder that it's not uncommon that teams are not aligned on priorities. While this feature was an important one to the users, we were constrained by limited resources. In hindsight, things may have been more efficient had Engineering been a part of our early alignment meetings or kickoff to help identify what was possible and what wasn't.
